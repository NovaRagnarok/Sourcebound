# Architecture

## Decision summary

This project starts as a **modular monolith with ports and adapters**.

Why:

- the domain is still moving
- GraphRAG, review, and truth modeling need tight iteration
- microservices would introduce orchestration cost too early
- Codex performs better on repos with explicit seams and low deployment complexity

## Canonical data flow

```text
Sources (Zotero)
  -> normalized text units
  -> candidate claims (GraphRAG)
  -> review decisions
  -> approved claims + evidence (Wikibase)
  -> retrieval projection (Qdrant)
  -> provenance-aware answers
```

## Domain layers

### Domain
Pure models and enums.
No framework imports.

### Ports
Interfaces for:
- source intake
- extraction
- candidate storage
- truth store
- vector projection

### Services
Use cases and orchestration.
Examples:
- ingest sources
- extract candidates
- approve claim
- answer query

### Adapters
Concrete implementations of ports.
Examples:
- file-backed dev storage
- Zotero API client
- GraphRAG runner
- Wikibase adapter
- Qdrant projection adapter

### API
FastAPI routes and dependency wiring.

## Storage strategy

### Day-one development
Use JSON files in `data/dev/`.

### Real system
- Zotero: source metadata + attachments
- Wikibase: approved claims and evidence references
- Qdrant: retrieval projection
- Postgres: job state, review state, app-specific state

## Why not store everything in Wikibase?

Because the system contains several kinds of data with different responsibilities:

- source-library concerns belong in Zotero
- canonical reviewed claims belong in Wikibase
- retrieval projections belong in Qdrant
- workflow/application state belongs in app storage

## Review gate

The review gate is the key trust boundary.

- candidate claims are generated by extraction
- approved claims are created only by explicit review decisions
- projections are generated only from approved claims

## Query policy

The query service should only answer from approved claims and linked evidence.

If evidence is weak or missing, the answer must surface the gap instead of smoothing it over.

## Initial future split points

Do not split yet, but preserve these seams:

- `corpus` module
- `extraction` module
- `review` module
- `canon` module
- `retrieval` module
- `query` module

If operational pressure grows, the first likely extractions are:

1. extraction worker
2. web UI
3. retrieval/query service
