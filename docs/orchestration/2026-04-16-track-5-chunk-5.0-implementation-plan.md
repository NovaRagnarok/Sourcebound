## Problem Statement

The trusted-operator auth boundary is now explicit, but the app still allows
anonymous access everywhere. This chunk should add the smallest real
authentication layer that removes anonymous write access and makes
unauthenticated versus unauthorized responses explicit, without widening into
full role-based route lockdown or UI authorization behavior from later chunks.

## Implementation Plan

1. Inspect the current API dependency layer, route structure, settings, and
   domain models for the smallest auth insertion point.
2. Define the minimum authenticated actor representation that fits the
   trusted-operator boundary without committing to a full RBAC design.
3. Protect non-public write or mutation surfaces with authentication only.
4. Add explicit unauthenticated and unauthorized response behavior in the API.
5. Add bounded integration tests that prove anonymous write access is gone,
   public reads still work, and the minimal auth layer does not spill into the
   finer-grained authorization rules reserved for 5.1.

## Acceptance Criteria

- anonymous write access is gone
- unauthenticated and unauthorized responses are explicit
- the implementation matches the trusted-operator auth boundary without role-rule drift

## Non-Goals

- no full role-rule matrix for operator-only and high-risk actions yet
- no UI authorization behavior changes
- no public sign-up, tenant provisioning, or internet-facing auth rollout
- no deployment secret-management work
