## Problem Statement

Chunk 5.0 removes anonymous write access with a single trusted-operator auth
gate, but it does not yet distinguish between writer-facing access and
operator-only runtime or administrative actions. This chunk should add the
smallest explicit role rules needed for high-risk routes and reflect those
authorization failures in both API and UI behavior, without widening into
provider selection, deployment hardening, or broader multi-user product work.

## Implementation Plan

1. Inspect the current auth dependency, settings, route surface, and operator
   UI fetch layer to identify the smallest viable writer-versus-operator split.
2. Define explicit role requirements for high-risk actions only.
3. Apply those role requirements to the bounded API routes in scope.
4. Reflect authorization failures in the operator UI without redesigning the
   broader auth model.
5. Add bounded API and UI-facing regression coverage for the role boundary.

## Acceptance Criteria

- high-risk routes require the intended role
- authorization failures are reflected in API and UI behavior
- the writer-versus-operator split stays bounded to the routes and UI affordances needed in this chunk
