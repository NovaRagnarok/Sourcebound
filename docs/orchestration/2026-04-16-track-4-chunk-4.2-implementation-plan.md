## Problem Statement

Sourcebound already supports Zotero, but it still reads as setup-dependent
rather than routine. The repo has a dedicated `zotero-check` command, live
integration coverage, and bounded adapter surfaces, which makes Zotero the
smallest optional integration that can realistically be moved from “supported”
to “routine” without widening into retrieval architecture or GraphRAG setup
heaviness.

## Implementation Plan

1. Inspect the current Zotero adapter, `zotero-check` diagnostics, live
   integration coverage, and top-level docs.
2. Identify the smallest changes needed to make Zotero setup, diagnostics, and
   the happy-path verification story feel routine.
3. Add or refine one smoke or verification path that proves the routine Zotero
   flow without widening into broader optional-integration work.
4. Align the docs so they describe Zotero as the single productized optional
   integration for this slice and explain how to verify it.
5. Run chunk-local verification, then prepare the chunk for read-only review.

## Acceptance Criteria

- Zotero setup is documented and tested
- operator diagnostics for Zotero are explicit
- a Zotero smoke or verification path exists
- no second optional integration is widened into scope
