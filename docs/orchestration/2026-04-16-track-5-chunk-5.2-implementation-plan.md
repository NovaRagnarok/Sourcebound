## Problem Statement

Sourcebound’s supported stack now has a clearer auth boundary and deployment
shape, but the repo still needs safer upgrade discipline for storage and
projection evolution. This chunk should add one bounded upgrade workflow and
one real verification path for schema or projection rebuild safety, without
widening into broader deployment hardening or release automation.

## Implementation Plan

1. Inspect the current storage, Postgres-backed projection, and CLI rebuild
   paths to find the smallest credible upgrade safety seam.
2. Add a bounded migration or compatibility discipline for the chosen seam.
3. Wire at least one operator-facing upgrade or rebuild workflow into the CLI
   or storage surface.
4. Document the supported upgrade steps in `docs/DEPLOYMENT.md`.
5. Add at least one upgrade-path or rebuild verification that proves the
   documented path works locally.

## Acceptance Criteria

- upgrade steps are documented
- at least one upgrade-path verification exists

## Verification Commands

- `.venv/bin/pytest -q tests/test_cli_qdrant.py tests/test_postgres_integration.py -k 'qdrant or rebuild or upgrade or projection'`
- `.venv/bin/pytest -q`

## Non-Goals

- no internet-facing deployment hardening or TLS/secret-management work
- no broad release automation redesign
- no auth-provider or role-boundary changes
- no schema churn outside the bounded upgrade seam selected for this chunk

## Reviewer Notes

Review only the migration and upgrade safety slice. Reject drift into 5.3
deployment hardening, broader release operations, or unrelated storage
refactors that do not directly support a documented upgrade or rebuild path.
