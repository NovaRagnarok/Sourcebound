## Problem Statement

The repo already names broad status buckets such as shipped, optional, and not
yet productized, but it does not yet present one crisp support matrix for
Python versions, backend combinations, and support levels. This chunk should
make the supported versus experimental versus provisional boundary explicit
without reopening deployment topology, auth, or optional-integration work.

## Implementation Plan

1. Inspect the current top-level runtime and deployment docs for version and
   backend support wording.
2. Add one compact support matrix or equivalent documentation structure that
   makes supported runtime modes explicit.
3. Align `README.md`, `docs/DEPLOYMENT.md`, and `docs/AUTHOR_STACK.md` so they
   use the same support-level language.
4. Run docs-focused verification that checks the matrix language is present and
   consistent.
5. Prepare the chunk for read-only review without widening into Track 5 auth or
