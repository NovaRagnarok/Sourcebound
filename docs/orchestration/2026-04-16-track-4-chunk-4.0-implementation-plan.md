## Problem Statement

Sourcebound already describes itself as a trusted-operator or local-first tool,
but the repo does not yet have an accepted ADR that fixes the near-term auth
boundary. Without that decision, later auth work could drift into unsupported
public multi-user assumptions or underspecify who should be authenticated, what
roles exist, and which surfaces must be protected first.

## Implementation Plan

1. Inspect the current wording in `README.md`, `docs/ROADMAP.md`, and
   `docs/DEPLOYMENT.md` for the existing trusted-operator boundary.
2. Land one ADR that defines the near-term identity model, the intended role
   boundary, and the surfaces that must become protected before broader
   internet-facing support is implied.
3. Align the top-level product and deployment docs so they reinforce that ADR
   and stop implying unsupported public or multi-tenant modes.
4. Run docs-focused verification that checks the boundary wording stays
   consistent across the ADR and the top-level docs.
5. Prepare the chunk for read-only review without widening into auth
   implementation, authorization rules, or deployment hardening.

## Acceptance Criteria

- a new ADR captures the boundary for identity, roles, and protected surfaces
- top-level docs stop implying unsupported auth or public multi-user modes
- no auth implementation or route behavior changes are introduced
