## Problem Statement

Sourcebound now has a clearer local and single-host story, plus bounded auth
and upgrade safety, but the repo still lacks one explicitly supported
internet-facing deployment shape for a small trusted team. This chunk should
validate exactly one such shape with truthful TLS/auth/secret-handling guidance
and an exercised end-to-end path, without widening into broad platform
engineering or multi-host deployment design.

## Implementation Plan

1. Inspect the current compose, infra, and deployment guide for the smallest
   supported internet-facing shape already closest to reality.
2. Add only the overlay compose and front-door pieces needed to make one
   TLS-terminated proxy shape explicit without disturbing the existing local or
   single-host stack.
3. Make the writer/operator token and secret-loading expectations explicit for
   that one shape.
4. Exercise that supported shape end to end in one bounded verification path,
   including one authenticated operator mutation through the public proxy.
5. Document the exact support boundary and non-goals in `docs/DEPLOYMENT.md`.

## Acceptance Criteria

- deployment guide is explicit about one supported TLS-terminated
  internet-facing shape, including writer/operator token handling and secret
  loading expectations
- the supported deployment is exercised end to end at least once through the
  public proxy, including one authenticated operator mutation

## Verification Commands

- `docker compose -f docker-compose.yml -f infra/compose/internet-facing.yml config`
- `docker compose -f docker-compose.yml -f infra/compose/internet-facing.yml up -d --build postgres qdrant app proxy`
- the bounded HTTPS health, workspace, and authenticated operator-intake proof
  for the internet-facing path

## Non-Goals

- no broad multi-host or autoscaled deployment architecture
- no public sign-up or tenant provisioning work
- no release-automation redesign beyond the bounded deployment proof
- no identity-provider, session, or per-user account work
- no unrelated app feature work outside deployment support

## Reviewer Notes

Review only the single supported internet-facing deployment slice. Reject drift
into broader infrastructure design, public multi-tenant assumptions, identity
provider work, or release-process work that is not required for the bounded
deployment proof.
