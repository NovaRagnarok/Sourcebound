## Problem Statement

`/health/runtime` and `saw status` already describe runtime state, but the repo still lacks one canonical verification command that says whether the recommended default stack is truly usable right now. Setup guidance is also incomplete for non-default backend choices, so operators can see `ready` while still being off the documented Postgres-plus-Qdrant path.

## Implementation Plan

1. Inspect the current runtime-status guidance, CLI status output, and setup-oriented tests.
2. Add the smallest shared helper(s) needed to describe default-stack setup requirements and verification failure conditions from one source of truth.
3. Add a canonical `verify-default-stack` CLI command that exits `0` only when the recommended local stack is genuinely usable.
4. Reuse shared guidance wording in CLI failure output instead of creating a separate setup vocabulary.
5. Add or update health tests to cover shared guidance language and verify-command success and failure behavior.
6. Run local verification for the chunk, then prepare the packet for review.

## Acceptance Criteria

- CLI status and `/health/runtime` use the same setup terminology for the core default-stack states.
- A canonical `verify-default-stack` command exists and exits `0` only when the default stack is genuinely usable.
- The verify command reuses the same runtime-status source of truth as CLI status and `/health/runtime`.
- Tests cover both the shared guidance language and the success/failure behavior of the new verify command.
- No unrelated files are edited.
