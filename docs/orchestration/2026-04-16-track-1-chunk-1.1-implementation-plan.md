## Problem Statement

The current runtime health model reports useful details, but `saw serve` still leaves too much inference work to the operator when required local services are unavailable or contradictory settings are present. This chunk is about the `saw serve` failure path only: make startup failure explicit and actionable before the server tries to run, without redesigning broader runtime or workspace wording.

## Implementation Plan

1. Inspect the current `serve` command path, startup validation, and runtime status messaging.
2. Identify the missing preflight checks for:
   - occupied app port
   - unavailable default services such as Postgres or Qdrant
   - contradictory configuration that should fail before startup
3. Add the smallest bounded preflight mechanism needed so `saw serve` emits direct, copy-pasteable guidance.
4. Add or update tests for at least one port-collision path and one missing-service path.
5. Run chunk-scoped verification first, then the strongest relevant broader verification for this slice.

## Acceptance Criteria

- Occupied-port detection is explicit in the `serve` path.
- The `saw serve` failure message includes the exact missing dependency and a concrete fix command.
- At least one port-collision test and one missing-service test exist.
- Chunk changes stay within the allowed files.
