## Problem Statement

The workspace already exposes setup actions and runtime status, but first-run operators still need to infer which items are blocking the default writing path and which integrations can wait. This chunk should tighten the workspace setup/degraded-state copy, badges, and action framing so the default stack reads as a clear required path while optional integrations stay visibly optional.

## Implementation Plan

1. Inspect the workspace summary API, first-run workspace rendering, and current setup/degraded-state styling.
2. Add the smallest route-level metadata or wording changes needed to classify blocking actions versus optional follow-ups.
3. Update the workspace UI copy and badges so first-run operators can identify what is blocking writing without leaving the workspace.
4. Ensure optional integrations such as Zotero, GraphRAG, and research semantics do not read like blockers for the default path.
5. Add or update API integration coverage for the setup summary behavior and run chunk-local plus repo-wide verification.

## Acceptance Criteria

- First-run operators can see what is blocking them from the workspace setup/degraded-state surfaces alone.
- Optional integrations do not look like blockers in the workspace setup/degraded-state surfaces.
- Workspace actions and setup cues clearly distinguish required steps from optional follow-up work in the setup/degraded-state surfaces.
- Tests cover the workspace summary guidance changes needed for the first-run state.
- No unrelated files are edited.
